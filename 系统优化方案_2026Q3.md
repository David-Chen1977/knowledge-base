> 基于 2026-06-25 全系统审计。分级：R0(止损) → R1(提效) → R2(重构) → R3(进化)。

## TL;DR — 最该做的三件事

**1. 清理尸体代码。** brain/ 目录不存在但5个脚本在引用；三件套输出/ 不存在但SYSTEM_CORE在引用；WorkBuddy已退役但4个脚本还在调用。每次开机读 SYSTEM_CORE 都在读假地址。

### R0.2 AgentMemory MCP — 要么启动，要么移除

### R0.4 网站部署最后一公里

当前流程：
```
npm run build ✅ → dist/ ✅ → git push → Cloudflare ❌
```
断在 CLOUDFLARE_API_TOKEN 缺失。创建一个真正的 deploy.sh：

```bash
#!/bin/bash
# deploy.sh — 一键构建+部署
set -e
cd /Users/Admin/OpencodeWorkspace/personal-website

# 从 Keychain 读取 token（同 wenyan-mcp 模式）
TOKEN=$(security find-generic-password -a 'opencode-human-collab' -s 'cloudflare-api-token' -w 2>/dev/null)
if [ -z "$TOKEN" ]; then
  echo "❌ 请在 Keychain 中设置 cloudflare-api-token"
  exit 1
fi

npm run build
CLOUDFLARE_API_TOKEN="$TOKEN" npx wrangler pages deploy dist --project-name chendaolei-website
echo "✅ 部署完成: https://chendaolei-website.pages.dev"
```

**Keychain 设置命令：**
```bash
security add-generic-password -a 'opencode-human-collab' -s 'cloudflare-api-token' -w '你的token'
```

**改动量：** 创建 deploy.sh，存一次 Keychain。30分钟。

## R1 — 提效（本周可做）

### R1.1 内容管线从"先写后发"改为"模板驱动"

当前痛点：每次写公众号需要手动处理 标题/字数/排版/QC/封面图 → 高频返工。

**方案：** 把公众号质量控制规范.md 编码为 `content-pipe.mjs` 的 preflight hook：

```
content-pipe.mjs run --preflight   # 自动检查标题字数/开篇钩子/分段长度
content-pipe.mjs run --wechat      # 从模板生成格式化的公众号版
content-pipe.mjs run --website     # 从同一内容生成 Astro .md
content-pipe.mjs run --publish     # 发布到草稿箱 + 构建网站
```

迁移 `qc-gate.mjs`（343L，现有但可能不工作）为真正的 preflight checker，检查：
- 标题 ≤15字 ✅/❌
- 字数 2000-3500 ✅/❌
- 开篇有钩子 ✅/❌ (检测是否为场景/对话/数据开头)
- 引用块 ≥1 ✅/❌
- 表格 ≥1 ✅/❌
- 免责声明 ✅/❌

**改动量：** 重构 content-pipe.mjs + qc-gate.mjs。4小时。

### R1.2 合并两个 PPT 管线

当前有两套 PPT 生成方式：
1. `nmg-ppt/native_pptx.py` — 手动写 deck.json → 生成 PPTX
2. `ppt-mcp` (Node.js) — 通过 MCP 工具生成

两套没有优劣之分，但双线并行增加认知负担。实际使用中 **nmg-ppt 更常用**。

**方案：** 删除 `ppt-mcp` MCP 配置，统一走 nmg-ppt。在 `内容输出/` 下建 PPT 模板目录，每次只需写 deck.json 中的数据部分。

**改动量：** 修改 opencode.jsonc + 建模板目录。30分钟。

### R1.3 建立 AnySearch 数据刷新脚本自动化

当前：`refresh-china-energy.sh` 是"伪脚本"——它只打印提示，没有实际执行 AnySearch 查询。

**方案：** 让 refresh-china-energy.sh 真正执行 AnySearch batch_search：

```bash
# refresh-china-energy.sh — 真正版
ANYSEARCH_URL="https://api.anysearch.com/mcp"

batch_query() {
  curl -s --max-time 30 --noproxy "*" -X POST "$ANYSEARCH_URL" \
    -H "Content-Type: application/json" \
    -d "$(cat)"
}

# 电价数据 → 输出表格
echo '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"search","arguments":{"query":"2026年6月各省电力现货市场均价"}}}' | batch_query
```

同时增加 launchd 定时任务：每天 9:00 + 21:00 自动刷新。

**改动量：** 重写 refresh-china-energy.sh + 配置launchd。2小时。

### R1.4 封面图管线自动化

当前每次发布公众号都需要手动准备封面图。我刚刚用 PIL 生成了一个，但字体路径需要修复。

**方案：** 修好 `generate_cover.py`（108L），让它从文章标题自动生成封面图：

```bash
python3 scripts/generate_cover.py "AI用电，账还没算" 内容输出/AI+能源双向赋能/cover.png
```

需要：修复字体路径（PingFang.ttc 在 macOS 上路径可能是 `/System/Library/Fonts/PingFang.ttc` 但 Pillow 不支持 ttc，需要用 ttf）。
- 用 `PIL.ImageFont.truetype("/System/Library/Fonts/Supplemental/Songti.ttc", 52)` 或
- 把 ttc 拆成 ttf

**改动量：** 修复 generate_cover.py 字体。1小时。

## R2 — 重构（2-4周）

### R2.1 从"手工管线"到"一键三件套"

当前状态：一个选题要经过 选题→素材→写作→QC→排版→发布 六个步骤，每个需要人工介入。启动一次三件套产出约需 2-3 小时。

**目标：** 一次触发，30分钟内完成三件套初稿。

**方案：** 创建 `trigger-publish.mjs`，接收一个选题描述，自动执行：

```
选题描述 → trigger-publish.mjs
  ├─→ AnySearch batch_search 采集素材 (3个并行查询: 政策/数据/对标)
  ├─→ LLM 生成公众号版 (按质控规范) → qc-gate.mjs 自检
  ├─→ LLM 生成网站版 (从公众号版扩写)
  ├─→ 生成 cover.png (调用 generate_cover.py)
  ├─→ 生成 deck.json → native_pptx.py → PPT
  ├─→ 发布公众号草稿箱 (wenyan-mcp)
  ├─→ 写入 Astro .md → npm run build
  └─→ 回流知识库 + 更新内容日历
```

每个子步骤可独立重试/跳过。失败时停在当前步骤，不阻塞后续。

**改动量：** 创建 trigger-publish.mjs (~500L)。后续迭代。1-2周。

### R2.2 知识库网站统一发布

当前有两个网站发布点：
1. **个人网站** (Astro → Cloudflare) — 博客文章，154页 ✅
2. **知识库网站** (VitePress → Vercel) — 赛道文件，DEPLOY_GUIDE.md 描述但状态不明

**方案：**
- 审计知识库网站是否还在更新
- 如果活跃：统一发布方式，用同一个 Keychain token 管理
- 如果不活跃：归档到个人网站的子目录 `/knowledge/`

**关键：** 两个网站用不同的框架（Astro vs VitePress）是合理的，但部署方式应该统一。

## R3 — 进化（Q3-Q4 roadmap）

### R3.1 MCP 数据层定时刷新

当前 MCP 数据（china-ets 碳价、fred 宏观、yfinance 行情）只在主动调用时更新。
知识库的"自动更新于"标签实际上需要手动触发。

**方案：**
```
launchd 定时任务 (每天 9:00 + 21:00):
  ├─ china-ets: get_market_summary → 更新 碳市场/赛道文件.md
  ├─ AnySearch: batch_search 电价/装机/政策 → 更新 赛道总览.md + 电力市场/赛道文件.md
  └─ fred: 宏观指标 → 更新 一级市场估值基准.md
```

配置文件：`crontab` 或 `launchd` plist。

### R3.2 内容表现跟踪闭环

目前内容产出后没有"反馈回路"：
- 文章发出去了 → 不知道多少人看了 → 不知道哪些观点被认可 → 下次写一样的

**方案：**
- 公众号阅读量/转发量 → 每周写入 `weekly-report.mjs`
- 哪些话题打开率高 → 反馈到 `topic-sense.mjs`
- 哪些判断被市场验证/证伪 → 回流知识库

需要：公众号后台数据导出（手动或 API）、分析仪表盘。

### R3.3 从"写文章"到"投资决策辅助"

当前系统的能力峰值是内容生产（三件套）。但核心业务是 PE/VC 投资。
下一步应该让系统直接辅助投资决策：

```
选题研究 → 自动生成：
   ├── 三件套（内容输出）
   ├── 项目DD清单（验证问题 → 自动检索 SEC/碳价/对标）
   ├── 估值模型（MCP数据 → 可比公司 → 估值区间）
   └── 交易简报（1页 PDF，含风险信号标注）
```

这依赖于 secedgar（已工作）、china-ets（已工作）、prospector-energy（已工作）的数据持续喂养。
